# 0.5.3-dev RAG Executor 修复规划

版本：0.5.3-dev
日期：2026-05-20
状态：规划完成，待 Codex 实施

---

## 1. 结论

0.5.2-dev 的 S50 RAG 反查仍标记为 `prepared_not_executed` / `requires_external_rag_operator`，要求用户手工提供回执。这是一个职责边界设计错误：RAG 导库（ingestion）确实不属于 ReferenceFootnote，但 RAG 反查（reverse lookup query）在 2.5 导库完成后应由 ReferenceFootnote 自主执行。0.5.3-dev 的核心修复是：**引入 RAG executor 层，使 S50 在 2.5 完成后默认自动执行反查，不再要求用户提供回执。**

---

## 2. 当前问题根因

| 根因 | 说明 |
|------|------|
| 边界过度泛化 | `boundaries.yaml` 中 `live_rag_lookup` 被列为 forbidden，但未区分"RAG 导库"和"RAG 查询"。导库是检索入库职责，查询是 ReferenceFootnote 自有职责。 |
| `build-post-ingestion-rag-call.py` 设计为"只生成包不执行" | 第 55 行 `"execution_status": "prepared_not_executed"` 和 `"requires_external_rag_operator": True` 硬编码了"等待外部操作者"语义。 |
| 无 executor 抽象 | 仓库中没有任何脚本负责"拿着 request 去调 RAG API 并写回 response"。所有流程都假设用户手工粘贴回执。 |
| `collaboration-flow.md` 步骤 8→9 之间缺少执行环节 | 步骤 8 生成 request，步骤 9 直接消费 response，中间的执行被隐式推给用户。 |
| `state-machine.md` S50 描述模糊 | 只写了"per-claim RAG response interpreted"，未说明 response 从哪来。 |

## 3. 0.5.3-dev 目标行为

### 3.1 正常路径（有 RAG 配置）

```
S45 intake quality gate passed
  → S50a: build-rag-request.py 生成 request
  → S50b: run-rag-reverse-lookup.py 读取 request + rag-executor-config，调用 RAG API，写入 response
  → S50c: validate-rag-response.py 消费 response
  → S55+: grounding → ledger → evidence map → ...
```

### 3.2 缺配置路径

```
S45 passed
  → S50a: request 生成成功
  → S50b: run-rag-reverse-lookup.py 检测到缺少 rag-executor-config
  → 输出 blocker: missing_rag_executor_config
  → flow-status 标记 blocked_at=S50, block_reason=missing_rag_executor_config
  → 不写 "requires_external_rag_operator"，不要求用户提供回执
```

### 3.3 离线测试路径

```
S45 passed (fixture)
  → S50a: request 生成
  → S50b: run-rag-reverse-lookup.py --mock 读取 fixture response，写入 state/rag-calls/<batch>.response.json
  → S50c: validate-rag-response.py 正常消费
```

### 3.4 职责边界不变项

- ReferenceFootnote 仍不得运行 CNKI、WoS、Zotero、PDF 获取、RAG 导库。
- ReferenceFootnote 仍不得操作 openclaw-cnki-takeover。
- RAG 导库完成回执仍由检索入库提供。
- ReferenceFootnote 只在 2.5 完成后执行 RAG 查询（read-only query），不修改知识库内容。

## 5. 推荐脚本/API/配置设计

### 5.1 新增脚本：`scripts/run-rag-reverse-lookup.py`

职责：读取 request，调用 RAG API，写入 response。

```
参数：
  --task-dir       任务目录
  --batch-id       批次 ID
  --mock           使用 mock executor（读取 fixture 而非调 API）
  --config-path    可选，覆盖默认配置路径

逻辑：
  1. 读取 state/rag-requests/<batch>.json
  2. 读取 RAG executor 配置（见 5.3）
  3. 若配置缺失 → 输出 blocker，return 1
  4. 若 --mock → 从 tests/fixtures/mocks/ 或 config 指定路径读取预制 response
  5. 否则 → 按 claim 逐条或批量调用 RAG API
  6. 写入 state/rag-calls/<batch>.response.json
  7. 输出 result JSON
```

### 5.2 修改脚本：`scripts/build-post-ingestion-rag-call.py`

- 移除 `"execution_status": "prepared_not_executed"` 和 `"requires_external_rag_operator": True`。
- 改为只生成 request metadata（保留 kb_context、return_contract）。
- 实际执行由 `run-rag-reverse-lookup.py` 负责。
- 或者：将此脚本重命名/合并到 `build-rag-request.py`，统一入口。

### 5.3 新增配置：`config/rag-executor.yaml`

```yaml
# RAG executor 配置
# ReferenceFootnote 在 S50b 使用此配置执行 reverse lookup query

executor:
  mode: mock                    # mock | live
  mock_response_dir: tests/fixtures/mocks/

live:
  base_url: null                # e.g. http://localhost:8080/v1
  api_key_env: RAG_API_KEY      # 从环境变量读取
  model: null                   # RAG 平台模型标识
  route: /chat/completions      # 或 /retrieve
  timeout_seconds: 60
  max_retries: 2

kb_routing:
  # 从 intake-status.json 的 kb_context 自动读取
  # 也可手动覆盖
  override_kb_ids: []

blocker_policy:
  # 缺少以下任一项时输出 missing_rag_executor_config blocker
  required_for_live:
    - base_url
    - api_key_env
    - model
```

### 5.4 response 写入位置

```
state/rag-calls/<batch>.response.json
```

格式与 `tests/fixtures/mocks/rag-service-mock.json` 一致（protocol_version=1.0, response_type=reverse_lookup_result）。

### 5.5 KB ID 读取逻辑

`run-rag-reverse-lookup.py` 从以下位置读取 KB IDs（优先级递减）：
1. `config/rag-executor.yaml` → `kb_routing.override_kb_ids`（若非空）
2. `state/intake-status.json` → `kb_routing.kb_ids`
3. `state/rag-calls/<batch>.json` → `kb_context.kb_ids`（兼容旧格式）

## 9. 是否允许 Codex 实施

**允许。** 本次修复：
- 不触及真实 CNKI/WoS/Zotero/PDF/RAG 导库
- 不操作 openclaw-cnki-takeover
- 不探测 localhost:22
- 不执行正式文章任务
- 不部署 staging/production
- mock 模式不触发真实 RAG API

风险点：
1. `live_rag_lookup` 从 forbidden 移除后，若 executor 配置被误设为 live 且有真实 API key，可能触发真实 RAG 查询。**缓解**：默认 mode=mock；live 模式需要 base_url + api_key_env 同时存在才激活。
2. 旧版 runtime（0.5.1-dev）不认识新配置文件。**缓解**：install-runtime.py 更新时覆盖；旧 runtime 不影响仓库开发。
3. golden-grid 任务目录已有 `rag-calls/golden-grid-sequential-01.json`（旧格式含 prepared_not_executed）。**缓解**：executor 写入 `.response.json` 后缀，不覆盖旧 call package；validate-rag-response.py 优先读 response.json。

**建议**：Codex 先执行 Phase 1-2（配置+脚本），运行 fixture gate 验证，再执行 Phase 3-5。
