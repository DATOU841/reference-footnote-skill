## 总体目标

0.4.0-dev 在 0.3.0-dev 已有的阶段机（A0-A11）和协同闭环基础上，强化六项核心能力：

1. **入库材料质量管控**：候选文献池的平均可消费正文级材料不低于 200 字/篇，低于池均值的个体文献标注风险。
2. **脚注本质回归**：脚注是正文内容的必要补充，不是参考文献罗列，引入脚注必要性审查门禁。
3. **脚注数量控制**：初始候选 15-25 个，经必要性裁剪后保留约 15 个最有价值脚注。
4. **参考文献数量控制**：初始候选池经筛选后保留 25-30 篇最重要参考文献，删除未消费文献。
5. **插入后真实性复核**：脚注插入位置和文献确定后，逐条复核真实性、准确性、一致性。
6. **脚注/尾注/参考文献边界厘清**：明确 footnote/endnote/reference_only 的类型区分和禁入规则。

**不变的硬边界**：不从零写论文、不替代正文写作、不直接运行 CNKI/WoS/Zotero/PDF/RAG、不探测 localhost:22、不操作 openclaw-cnki-takeover。所有外部操作仍通过结构化请求/回执完成。

## 新增原则

### 原则 1：入库材料质量原则

**核心规则**：入库候选文献池的平均可消费文本量不低于 200 字/篇（中文字符或等效英文 token）。这是一个**池级比率指标**，不是对单篇文献的硬性下限。

**设计意图**：

- 学术文章的参考文献来源质量参差不齐，部分法条、判例、短篇评论天然文本量少，不应一刀切拒绝
- 但如果整体候选池的平均可消费文本过低，说明入库材料普遍空洞，无法有效支撑脚注
- 200 字/篇的池均值确保大多数文献有足够正文级片段可供 RAG 命中和脚注撰写

**"可消费文本"定义**：

- 不是文献题录（title + authors + journal + year）
- 不是摘要式空话（"本文认为…本文试图…"）
- 是可供 RAG 命中、人工判断、脚注撰写的正文级材料：核心论证片段、关键数据描述、制度说明、概念界定、案例分析、法条解释等
- 来源可以是 RAG snippet、PDF 摘录、人工录入文本

**检查机制**：

- `intake-completion` 结果中新增 `usable_text_chars` 字段（整数），由检索入库为每条 result 返回
- ReferenceFootnote 的 `apply-intake-completion.py` 收集所有 result 的 `usable_text_chars`，计算池均值
- **池级门禁**：`avg_usable_text_chars = sum(usable_text_chars) / count(results)`，必须 ≥ 200
- **个体标注**（非阻断，仅标注风险）：
  - `usable_text_chars < 50`：标记 `material_flag: very_low`，建议补充或替换
  - `usable_text_chars` 在 50-149 之间：标记 `material_flag: below_average`，限制用于关键论断支撑
  - `usable_text_chars >= 150`：标记 `material_flag: normal`
- 个体文献不因 `usable_text_chars` 低而被自动拒绝，但池均值不达标时整批进入补库/人工复核

**池均值不达标时的处理**：

1. 质量报告中记录 `pool_avg_usable_text_chars` 和 `pool_material_status: insufficient`
2. 标记为 **warning**（非 blocking）：允许继续流程但在交付包中突出提示
3. 生成补库建议：列出 `material_flag: very_low` 的文献，建议检索入库补充正文材料
4. 进入人工复核清单：提示用户整体材料质量偏低，需人工判断是否补库

**个体文献的风险传递**：

- `material_flag: very_low` 的文献在 A9a 候选池中降低 `necessity_score`（材料质量因子得分低）
- `material_flag: below_average` 的文献不得单独支撑 `critical` claim，但可作为 `recommended` claim 的背景脚注
- 这些是 soft constraint，不是 hard rejection——最终由 necessity_score 排序和裁剪阶段综合决定

### 原则 2：脚注本质原则

**核心规则**：脚注是对正文内容的必要补充，不是"参考文献脚注"。

**禁止行为**：

| 禁止类型 | 说明 | 检测方法 |
| --- | --- | --- |
| 参考文献罗列 | 把文献题录直接变成脚注正文 | 检查脚注正文是否仅含题录格式文本 |
| 背景堆砌 | 用脚注堆砌无直接补充价值的背景 | 检查 `annotation_purpose` 是否为 `background` 且无具体内容 |
| 空泛废话 | "关于此问题，参见某某某" 式无内容脚注 | 检查脚注正文有效信息量 |
| 正文重复 | 脚注内容与正文已说清楚的内容重复 | 对比脚注文本与对应段落文本相似度 |
| 硬塞文献 | 无直接补充价值的文献被强行写入脚注 | 检查 `necessity_score` 和 `support_strength` |

**脚注合法用途**：

| 用途 | `annotation_purpose` 值 | 示例 |
| --- | --- | --- |
| 必要补证 | `evidence` | 为正文关键判断提供精确来源和页码 |
| 概念说明 | `clarification` | 解释正文使用的专门概念、术语界定 |
| 制度/史实/争议背景 | `supplement` | 正文不宜展开但必须交代的制度背景 |
| 关键来源精确支撑 | `source_anchor` | 对核心理论、权威判例的精确出处 |
| 对立观点标注 | `counter_view` | 标注正文论断的反对意见出处 |

### 原则 3：脚注数量控制原则

**目标数量**：每篇文章最终约 15 个脚注。

**流程**：

```
初始候选池 (15-25)
    ↓ 必要性审查
    ↓ 删除不必要/重复/空泛/弱支撑/背景堆砌
最终保留池 (≈15)
```

**必须优先保留的脚注**（即使超出 15 个也应保留）：

- 对 critical claim 的唯一支撑来源
- 对核心概念/定义的精确界定
- 对关键争议的权威出处
- 法条、判例、一手材料的精确引证
- 正文已明确提及但未给出处的关键来源

**必须优先删除的脚注**：

- `annotation_purpose: background` 且正文已有足够背景
- 同一 claim 有多个脚注时，留最强删最弱
- `support_strength: partial_support` 且有 `strong_support` 替代
- 仅堆砌 "参见某某" 无实质内容
- `material_flag: very_low` 且非 critical claim 唯一支撑

**数量异常处理**：

- 最终 < 10 个：警告 `footnote_count_low`，建议检查是否有遗漏的 critical claim
- 最终 > 20 个：阻断 `footnote_count_excessive`，必须再次裁剪
- 最终 10-20 个：通过，15 ± 5 为正常范围

### 原则 4：参考文献数量控制原则

**目标数量**：每篇文章最终保留 25-30 篇参考文献。

**保留优先级**：

| 优先级 | 类型 | 说明 |
| --- | --- | --- |
| P0 必保 | 核心理论/经典文献 | 正文核心论证直接依赖 |
| P0 必保 | 一手材料 | 法条、判例、档案、统计数据 |
| P1 强保 | 关键争议文献 | 正文争议部分直接引用 |
| P1 强保 | 深度消费文献 | 脚注中有实质性引用的文献 |
| P2 可保 | 背景参考 | 提供制度/历史背景的文献 |
| P3 可删 | 弱相关 | 仅在候选池中出现但未被正文/脚注实际消费 |
| P3 可删 | 重复 | 同一论点有多篇文献支撑，保留最权威者 |
| P3 必删 | 未消费 | 参考文献表中存在但正文和脚注均未实际引用 |

**参考文献数量异常处理**：

- 最终 < 20 篇：警告 `reference_count_low`
- 最终 > 35 篇：警告 `reference_count_high`，建议继续裁剪
- 最终 > 40 篇：阻断 `reference_count_excessive`

### 原则 5：插入后真实性复核原则

**核心规则**：脚注插入完毕、参考文献整理完毕后，必须逐条复核。

**复核维度**：

| 复核项 | 说明 | 生成方 | 验证方 |
| --- | --- | --- | --- |
| 插入位置正确性 | 脚注标记位于正确句子/段落 | ReferenceFootnote | 人工 |
| 脚注内容相关性 | 脚注确实补充正文，非无关废话 | ReferenceFootnote | ReferenceFootnote + 人工 |
| 文献真实性 | 文献真实存在，非伪造 | ReferenceFootnote 生成请求 | 检索入库/人工 |
| 题录准确性 | 题名、作者、来源、年份、页码准确 | ReferenceFootnote 生成请求 | PDF 核验/人工 |
| PDF 支撑确认 | 原 PDF 中确实包含脚注引用内容 | ReferenceFootnote 生成请求 | 外部 PDF 验证 |
| RAG-PDF 一致性 | RAG 命中片段与 PDF 原文一致 | ReferenceFootnote 生成请求 | 外部双核验 |
| 支撑充分性 | 脚注对正文 claim 的支撑是否充分 | ReferenceFootnote | ReferenceFootnote + 人工 |
| 页码/OCR 准确性 | 页码是否缺失/错误，OCR 是否可靠 | ReferenceFootnote | 外部验证 |
| 断章取义检测 | 引用是否脱离原文语境 | ReferenceFootnote 生成请求 | 人工 |
| 参考文献表一致性 | 参考文献表与脚注/正文实际消费一致 | ReferenceFootnote | ReferenceFootnote |

**ReferenceFootnote 的角色**：

- **生成**复核请求（`authenticity-verification-request.json`）
- **接收**复核结果（`authenticity-verification-result.json`）
- **验证**结构化回执的完整性和一致性
- **输出**问题清单（`authenticity-issues.json`）
- **不直接**获取 PDF、不直接跑 RAG、不直接核验真实性

### 原则 6：脚注/尾注/参考文献边界厘清原则

**当前问题诊断**：

0.3.0-dev 中 `plan-footnotes.py` 将所有支撑证据统一生成为 `insertions[]`，未区分：
- 脚注（页脚/文末补充说明）
- 尾注（与脚注功能相同但呈现在文末，仅为排版形式差异）
- 参考文献引用（仅标注来源，不含补充说明文本）

**新增字段**：

```json
{
  "note_type": "footnote | endnote | reference_only",
  "annotation_purpose": "evidence | clarification | supplement | source_anchor | counter_view | background | reference_only"
}
```

**规则**：

| `note_type` | 正文中呈现 | 脚注正文 | 参考文献表 |
| --- | --- | --- | --- |
| `footnote` | 脚注标记 | 有补充说明文本 | 对应文献入表 |
| `endnote` | 尾注标记 | 有补充说明文本（呈现在文末） | 对应文献入表 |
| `reference_only` | 参考文献序号 | **无脚注正文** | 对应文献入表 |

**禁入规则**：

- `annotation_purpose: reference_only` 禁止进入脚注正文（即不得为 `note_type: footnote` 或 `endnote`）
- 如果一个 insertion 的实质只是标注来源、无补充说明价值，必须降级为 `reference_only`
- `note_type` 默认为 `footnote`；用户可配置为 `endnote`（仅排版差异）

### A8 扩展细节

**新增验证**：

```python
# apply-intake-completion.py 扩展
all_chars = []
for idx, item in enumerate(completion.get("results", []), start=1):
    chars = item.get("usable_text_chars", 0)
    all_chars.append(chars)
    if chars < 50:
        item["material_flag"] = "very_low"
        warnings.append(f"results[{idx}] usable_text_chars={chars}, flagged very_low")
    elif chars < 150:
        item["material_flag"] = "below_average"
    else:
        item["material_flag"] = "normal"

avg_chars = sum(all_chars) / len(all_chars) if all_chars else 0
pool_material_status = "sufficient" if avg_chars >= 200 else "insufficient"
completion["pool_avg_usable_text_chars"] = round(avg_chars, 1)
completion["pool_material_status"] = pool_material_status
if pool_material_status == "insufficient":
    warnings.append(f"pool avg usable_text_chars={avg_chars:.1f} < 200, material quality low")
```

**向下传递**：`material_flag` 随 result 传入 evidence map 重建；`pool_material_status` 进入 `quality-report.json`。个体 `material_flag: very_low` 的文献在 A9a 中降低 `necessity_score` 的材料质量因子。

### A9a 候选脚注池

**输入**：`evidence-map.json`、`article-structure.json`、`insertion-plan.json`（0.3.0 式原始版）

**输出**：`footnote-candidate-pool.json`

**逻辑**：

1. 遍历所有 `evidence_status` 为 `strong_support` 或 `partial_support` 的 claim
2. 对 `material_flag: very_low` 的文献不做硬性排除，但在 `necessity_score` 的材料质量因子中给予低分
3. 为每个候选生成：
   - `candidate_id`
   - `claim_id`、`paragraph_id`、`sentence_id`
   - `note_type`：默认 `footnote`；如果仅标注来源无补充说明，设为 `reference_only`
   - `annotation_purpose`：根据 claim_type 和 evidence_type 自动推断
   - `necessity_score`：0-100 分，综合 claim 重要性、证据强度、材料质量、唯一性
   - `footnote_draft`：脚注草稿文本
   - `supporting_ref_id`
   - `material_flag`：从 intake completion 传递的 `very_low` / `below_average` / `normal`
   - `usable_text_chars`
   - `risks[]`
4. 按 `necessity_score` 降序排列
5. 数量应在 15-25 之间；不足 15 则警告缺口

**`necessity_score` 计算规则**：

| 因子 | 权重 | 说明 |
| --- | --- | --- |
| `need_level` | 30% | critical=100, important=70, recommended=40 |
| `support_strength` | 25% | strong=100, partial=60 |
| `material_quality` | 20% | normal=100, below_average=50, very_low=15 |
| `uniqueness` | 15% | 该 claim 是否只有此一个支撑来源 |
| `annotation_richness` | 10% | 脚注草稿是否有实质补充内容（非仅题录） |

### A9b 脚注必要性裁剪

**输入**：`footnote-candidate-pool.json`

**输出**：`footnote-pruning-result.json`

**裁剪规则**（按优先级从高到低执行）：

1. **reference_only 禁入**：`annotation_purpose: reference_only` 的候选从脚注池移出，转入纯参考文献
2. **材料极低优先删除**：`material_flag: very_low` 且 `need_level != critical` 的候选优先进入裁剪队列（非硬性拒绝，但在同等条件下优先删除）
3. **重复裁剪**：同一 claim 有多个脚注时，保留 `necessity_score` 最高者
4. **空泛检测**：脚注草稿文本 < 30 字或仅含"参见"+"题录"格式时，标记为 `vacuous`
5. **背景堆砌检测**：`annotation_purpose: background` 且 `need_level != critical` 时降低优先级
6. **正文重复检测**：脚注文本与对应段落文本高度相似时，标记为 `redundant`
7. **数量裁剪**：从最低 `necessity_score` 开始删除，直到总数 ≤ 目标数量

**输出结构**：

```json
{
  "retained": [...],
  "pruned": [
    {
      "candidate_id": "...",
      "deletion_reason": "vacuous | redundant | duplicate | background_excess | material_very_low | reference_only_demoted | low_necessity",
      "necessity_score": 35,
      "original_rank": 18
    }
  ],
  "final_count": 15,
  "pruning_summary": {
    "initial_count": 22,
    "pruned_count": 7,
    "reasons": {"vacuous": 1, "duplicate": 2, "background_excess": 2, "low_necessity": 2}
  }
}
```

### A9c 参考文献裁剪

**输入**：`footnote-pruning-result.json`（确定哪些脚注保留）、`evidence-map.json`

**输出**：`reference-pruning-plan.json`

**逻辑**：

1. 收集所有出现在保留脚注中的文献 → `consumed_by_footnote`
2. 收集正文中明确提及的文献 → `consumed_by_text`
3. 计算每篇文献的 `reference_importance_score`：

| 因子 | 权重 | 说明 |
| --- | --- | --- |
| 被脚注消费次数 | 35% | 被多少个保留脚注引用 |
| 被正文消费深度 | 25% | 正文中提及程度 |
| 证据类型 | 20% | 一手材料/核心理论 > 背景参考 |
| 来源角色 | 10% | 理论锚点/概念界定 > 旁证补充 |
| 材料质量 | 10% | normal > below_average > very_low |

4. 按 `reference_importance_score` 排序
5. 保留前 25-30 篇
6. 未被正文或脚注实际消费的文献：标记为 `unconsumed`，默认删除，除非用户明确要求保留
7. 生成删除理由

**输出结构**：

```json
{
  "retained_references": [...],
  "pruned_references": [
    {
      "ref_id": "...",
      "title": "...",
```

### A10b PDF + RAG 双核验

**输入**：`authenticity-verification-result.json`（由外部返回）

**输出**：更新后的 `insertion-plan.json`、`authenticity-issues.json`

**逻辑**：

1. 验证回执结构完整性
2. 遍历每条 check 结果：
   - `passed`：标记为已核验
   - `failed`：根据 `check_type` 决定阻断级别
   - `uncertain`：进入人工复核
3. 阻断规则：

| check_type | failed 时动作 |
| --- | --- |
| `source_existence` | **阻断**：删除该脚注，标记为伪造风险 |
| `metadata_accuracy` | **阻断**：修正或删除 |
| `pdf_content_match` | **阻断**：标记 `pdf_mismatch`，进入人工复核 |
| `rag_pdf_consistency` | **警告**：标记 `rag_pdf_inconsistent`，进入人工复核 |
| `insertion_position` | **阻断**：修正插入位置或删除 |
| `footnote_relevance` | **警告**：标记为可能无效脚注 |
| `support_sufficiency` | **警告**：标记为弱支撑 |
| `context_integrity` | **警告**：标记 `possible_misquote` |

### A10c 脚注/参考文献一致性门禁

**输入**：最终 `insertion-plan.json`、`reference-pruning-plan.json`

**输出**：`consistency-gate-result.json`

**检查项**：

1. 每个脚注引用的文献是否在参考文献表中
2. 参考文献表中的文献是否至少被一个脚注或正文段落消费
3. 脚注编号是否连续、无跳号
4. 脚注 `note_type` 与实际内容是否一致（声称 `footnote` 但无补充文本 → 降级为 `reference_only`）
5. 无"幽灵引用"：参考文献表有但正文/脚注均未提及的文献
6. 无"悬空脚注"：脚注引用了不在参考文献表中的文献

**一致性问题分级**：

- `ghost_reference`：阻断（参考文献表有但无处引用）
- `dangling_footnote`：阻断（脚注引用了不在参考文献表中的来源）
- `number_discontinuity`：阻断（脚注编号不连续）
- `type_mismatch`：警告（`note_type` 与内容不符）

## 与检索入库的协同

### 新增协议要求

1. **intake-completion 新增字段**：`usable_text_chars`、`usable_text_source`

   检索入库返回每条 result 时，应计算该文献已入库的可消费文本长度。如果检索入库暂不支持该字段，ReferenceFootnote 默认标记为 `material_flag: unknown`，进入人工复核，池均值计算跳过该条。

2. **池均值不达标时的补库建议**：

   如果候选池的 `pool_avg_usable_text_chars < 200`，ReferenceFootnote 在质量报告中输出 warning，并生成补库建议，列出 `material_flag: very_low` 的文献，要求检索入库：
   - 补充这些文献的正文材料（如获取 PDF 全文、手动录入关键段落）
   - 或替换为同主题但材料更完整的文献
   - 目标是使池均值回升至 ≥ 200 字/篇

3. **真实性复核请求**：

   `authenticity-verification-request.json` 中 `check_type: source_existence` 和 `check_type: metadata_accuracy` 的验证可转交检索入库执行（它有数据库和 PDF 访问能力）。

## 边界与禁止事项

### 不变的硬边界

- 不从零写论文
- 不替代正文写作
- 不直接运行 CNKI、WoS、Zotero、PDF 获取、RAG 导库
- 不探测 localhost:22

- 不运行 writing-pool、advance-pool、mimo
- 不使用正式文章任务目录作为测试场

### 0.4.0-dev 新增边界

- **不直接验证真实性**：ReferenceFootnote 生成复核请求但不执行。不下载 PDF、不查询数据库、不访问出版社网站。
- **不修改正文内容**：只决定脚注和参考文献的增删，不修改正文段落文本。
- **不覆盖人工决策**：池均值不达标时建议补库，但人工可覆盖决策；个体文献低于均值时通过 `necessity_score` 软调节而非硬拒绝。
- **不伪造必要性评分**：`necessity_score` 基于可量化因子计算，不允许硬编码通过。
- **不跳过裁剪阶段**：A9b 和 A9c 是必经阶段，不允许直接从 A9a 跳到 A9 final。

### 禁止事项清单

| 禁止事项 | 原因 |
| --- | --- |
| 把参考文献题录作为脚注正文 | 违反脚注本质原则 |
| 未经裁剪的候选池直接进入最终方案 | 违反数量控制原则 |
| `material_flag: very_low` 的文献用于 critical claim 唯一支撑而不标注风险 | 违反材料质量原则 |
| 跳过真实性复核请求阶段 | 违反真实性复核原则 |
| 参考文献表含未消费文献（未标注理由） | 违反一致性原则 |
| `reference_only` 进入脚注正文 | 违反边界厘清原则 |
| 脚注超过 20 个不裁剪 | 违反数量控制原则 |
