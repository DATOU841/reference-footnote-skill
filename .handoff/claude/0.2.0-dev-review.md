### 边界审计

针对本轮六项审查目标逐项确认：

#### 1. evidence gap → 检索入库调用包

通过。`build-evidence-map.py` 将 `no_support` + `critical/important` 的 claim 识别为 `critical_gaps`；`build-search-handoff.py` 将 gap 转为完整的 `search-intake-requests/<batch>.json`（含 gap_id、claim_id、search_strategy 等全部协议字段）；`build-search-intake-call.py` 再封装为带有 `execution_status: prepared_not_executed` 的正式调用包。数据链路：evidence-map → search-intake-requests → search-intake-calls，逐层可溯。

#### 2. 中文调用提示词

通过。`build-search-intake-call.py` 的 `render_prompt()` 生成 `<batch>.prompt.md`，内容包含中文硬边界声明、输入包摘要、期望返回字段列表和读取指引。fixture_17 验证了 prompt 文件存在。

#### 3. 真实操作只能由检索入库在用户授权后执行

通过。多层声明：
- SKILL.md 第 19 行："真实检索和入库只能通过结构化请求交给 `检索入库`。"
- 调用包 JSON：`requires_user_authorization_for_real_search: true`，`allowed_real_executor: "检索入库"`，`forbidden_for_referencefootnote` 列出六类禁止操作。
- 生成的提示词第一条硬边界："只有在用户明确授权真实检索/入库时才执行。"
- `collaboration-flow.md` Boundary 段落明确列出七项 ReferenceFootnote must not。
- `config/boundaries.yaml` forbidden 列表完整。
- `startup.py` 在 `boundaries_blocked` 中阻断 cnki_wos_zotero_pdf_rag、openclaw_cnki_takeover、localhost_22_probe。

#### 4. 补库后二轮 RAG 调用只针对 rag_indexed=true

通过。`build-post-ingestion-rag-call.py` 第 30 行：`if not import_status.get("rag_indexed"): continue`。fixture_18 用含 `rag_indexed: true` 的 mock 验证了 targets 非空。`fixture-intake-completion-full.json` 提供了 `rag_indexed: true/false` 的混合场景（3 条结果，仅第 1 条 `rag_indexed=true`），fixture_16 验证其被正确记录。

#### 5. 未伪装为已完成真实检索/RAG

通过。所有新增脚本的输出 JSON 均标记 `execution_status: prepared_not_executed` 或 `requires_external_rag_operator: true`。`collaboration-flow.md` 明确规定 "ReferenceFootnote may say 'call package prepared' but must not say search or RAG has completed until the responsible system returns a structured completion or response."。无任何路径将 `prepared_not_executed` 自动翻转为 `completed`。

#### 6. 0.1.0-dev 边界和质量门禁保留

通过。原有 A0-A11 阶段机未被修改，新增 A7.5 和 A8.5 作为纯调用包准备阶段。质量门禁阈值（critical_claim_coverage ≥ 0.8、high_risk_citation_ratio ≤ 0.2、page_missing_ratio ≤ 0.3）保持不变。PROTECTED_TYPES、SUPPORT_STRENGTHS、RISK_FLAGS 全部保留。page_missing 自动注入逻辑保留。no_insert_zones 逻辑保留。anti-fabrication 保障（agents + docs + SKILL.md）保留。

### P2-3：coordinator / rag-interpreter agent 提示词极简

`agents/coordinator.md` 和 `agents/rag-interpreter.md` 各只有 1-2 句话，未包含 0.2.0-dev 协同阶段（A7.5、A8.5）的任何指引。当 agent 被 LLM 消费时，缺乏协同层操作说明可能导致理解不完整。建议在下一版本丰富为完整操作指南。

### P2-4：补库后二轮 RAG → 证据重建闭环未实际演练

fixture_18 验证了 `build-post-ingestion-rag-call.py` 生成调用包，但没有 fixture 演练「RAG 平台返回二轮结果 → validate-rag-response → rebuild evidence-map → rebuild footnote plan」的完整闭环。`collaboration-flow.md` 第 9-10 步描述了这个路径，但 fixture 覆盖止步于调用包准备。这是 0.1.0-dev P2-5（多轮闭环未演示）的延续，建议 0.3.0 增加端到端 fixture。

### P2-5：`quality-gates.md` 未更新 fixture 数量

文档仍写 "expects 15 offline fixtures to pass"，实际为 18 个。

### P2-6：调用包中 `allowed_server_entry_if_authorized` 字段

`build-search-intake-call.py` 第 88 行在调用包 JSON 中包含 `allowed_server_entry_if_authorized: "ssh ubuntu@beijing"`。这是一个面向 `检索入库` 的元数据字段，语义上合理（告知下游 skill 授权服务器入口）。但该字段在 `docs/collaboration-flow.md` 和 `docs/handoff-protocol.md` 中均未文档化。建议补充文档说明，或在调用包 schema 中标注其为可选参数。
