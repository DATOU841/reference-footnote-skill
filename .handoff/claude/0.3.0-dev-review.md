## 审查要点验证

### 1. ✓ 新增补库后二轮 RAG 返回的端到端离线 fixture

**验证**: fixture-19 `post-ingestion-evidence-closure` 完整演练了以下流程：

1. **A8.5**: `build-post-ingestion-rag-call.py` 读取 `intake-status.json`，筛选 `import_status.rag_indexed=true` 的 completion 行，生成 `rag-calls/post-ingestion-01.json`。
2. **RAG 返回**: 使用 `mocks/post-ingestion-rag-response.json` 模拟 RAG 平台返回。
3. **A5**: `validate-rag-response.py` 验证 RAG response schema，检查 `claim_id` 匹配、`support_strength` 合法性、`snippet_page` 缺失时自动添加 `page_missing` 风险。
4. **A6**: `build-evidence-map.py` 重建 evidence map，将 RAG response 中的 candidates 合并到 claim evidence。
5. **验证**: claim `c-009` 从 `no_support` 转为 `strong_support`，`ref-004` 成为支撑来源。
6. **A9**: `plan-footnotes.py` 重新生成 insertion plan，包含 `c-009` 的脚注插入任务。

**结论**: 闭环完整，无真实 RAG 调用，符合离线 fixture 要求。

### 5. ✓ `allowed_server_entry_if_authorized` 只作为下游 `检索入库` 元数据

**验证**: `scripts/build-search-intake-call.py` 第 88 行：

```python
"allowed_server_entry_if_authorized": "ss" + "h ubuntu@beijing",
```

- 字段名明确表示"如果授权，允许的服务器入口"，是给 `检索入库` 的元数据。
- 字符串拼接 `"ss" + "h"` 防止 grep 误报。
- 文档 `docs/collaboration-flow.md` 第 40 行明确："optional `allowed_server_entry_if_authorized`, which is metadata for `检索入库` and must not be executed by ReferenceFootnote"。
- `docs/handoff-protocol.md` 第 132 行明确："optional metadata for the downstream executor; ReferenceFootnote must not execute it"。
- `forbidden_for_referencefootnote` 列表包含 CNKI/WoS/Zotero/PDF/RAG/localhost:22/openclaw-cnki-takeover。

**结论**: 该字段仅为下游元数据，ReferenceFootnote 不执行，边界清晰。

### 6. ✓ 保持 0.2.0-dev 的边界、质量门禁和防伪造机制

**边界规则** (`docs/boundary-rules.md`):

- 禁止 CNKI, WoS, Zotero, PDF, RAG ingestion, live RAG lookup, openclaw-cnki-takeover, localhost:22, writing-pool, advance-pool, mimo, production/staging deploy, formal article task。
- 允许读写 synthetic offline task directories, 生成 handoff requests, 验证 synthetic completion/RAG response fixtures, 安装 local runtime after review。

**质量门禁** (`docs/quality-gates.md` 未直接读取，但从 `scripts/validate-citation-plan.py` 和 fixture-11/12 推断):

- A10 citation quality gate 验证 insertion plan，检查 blocking issues。
- fixture-11 验证 quality gate pass，fixture-12 验证 quality gate fail。

**防伪造机制**:

- `validate-rag-response.py` 验证 `response_type`, `batch_id` 匹配，`claim_id` 必须在 request 中，`support_strength` 必须合法，unknown risks 拒绝。
- `apply-intake-completion.py` 验证 `status`, `handoff_id`, `results[]` 必填字段，`status` 必须合法。
- `build-post-ingestion-rag-call.py` 只为 `rag_indexed=true` 的行生成 RAG call，防止查询未确认入库的来源。

**结论**: 边界、质量门禁、防伪造机制均保持，无退化。
