| # | 问题 | 表现 | 后果 |
|---|------|------|------|
| D1 | 无单线主状态源 | `state/status.json` 只记录启动状态，不记录当前阶段、已完成阶段、阻断原因 | 无法从任意中断点恢复；Codex 必须人工判断"现在到哪了" |

- `quality-report.json` 列出 blocking issues 后，流程直接进入 delivery。
- 没有"删除不能支撑的注释"、"降级过强表述"、"移动位置不合适的注释"、"删除未消费参考文献"的阶段。
- 没有"清理后重建 citation needs / 注释 / 参考文献"的阶段。
- 没有"清理后全文顺序核查"的阶段。

### 1.4 RAG 反查粗粒度

- `build-rag-request.py` 按 claim 生成请求，但 claim 是按主题分组的，不是按全文顺序逐段/逐句排列。
- RAG 返回结果没有强制记录段落 ID、句子 ID、章节上下文。
- 没有逐段/逐句的 RAG 反查循环。

### 1.5 证据链无 mimo-style 全程记录

- 正文写作 skill 有 mimo 暗链全程记录每条证据。
- ReferenceFootnote 没有等价能力：证据散落在 evidence-interpretations、evidence-map、insertion-plan、grounding-resolution 中。
- 没有按全文顺序的统一 ledger。

### 1.6 写作池能力缺失

- SKILL.md 声明了 A9d/A9e，但脚本不存在。
- 模板 `writing-pool-review-request.template.json` 和 `writing-pool-review-result.template.json` 存在，但无脚本消费。
- `docs/writing-pool-review-protocol.md` 存在且设计合理，但无实现。

### 1.7 硬门禁不足

| 风险类型 | 当前处理 | 应有处理 |
|----------|----------|----------|
| analogy_only 被当 strong_support | plan-footnotes.py 不生成 insertion（正确） | 但 evidence-map 中没有显式阻断标记 |
| 一手权属缺失 | plan-footnotes.py 不生成 insertion（正确） | 但没有在风险清单中强制标记为"必须清理" |

### 1.8 最终交付 gate 不够强

- `stage-gate.py` 和 `run-local-gate.py` 只检查结构和 fixture 通过。

必须检测的风险类型：

| 风险类型 | 严重度 | 检测逻辑 |
|----------|--------|----------|
| `ownership_unverified` | blocking | claim_type 含权属/产权/来源 + 无一手材料 |
| `direct_experiment_missing` | blocking | 教学效果 claim + 无直接实验数据 |
| `secondhand_citation` | blocking | RAG 返回的 chunk 引用了另一来源而非原始研究 |
| `page_missing` | warning→blocking | 最终交付时仍无页码 |
| `ocr_uncertain` | warning | grounding 有 OCR 风险 |
| `concept_approximate` | warning | 概念近似但不精确匹配 |
| `unconsumed_reference` | blocking | 参考文献未被正文或注释消费 |
| `analogy_as_strong` | blocking | analogy_only 被误标为 strong_support |
| `rag_claim_mismatch` | blocking | RAG 召回与正文 claim 语义不契合 |
| `writing_pool_rewrite_pending` | blocking | 写作池要求整段重写但未处理 |
| `writing_pool_drop_pending` | blocking | 写作池要求删除注释但未处理 |
| `no_pdf_no_markdown` | warning | 文献无 PDF 也无可消费 Markdown |

## 五、RAG 逐段/逐句反查机制设计

### 5.1 当前问题

`build-rag-request.py` 按 claim 生成 RAG 请求，但：
- claim 按主题分组，不按全文顺序。
- 没有段落 ID、句子 ID、章节上下文。
- 没有逐段/逐句循环。

### 5.2 新设计

#### 5.2.1 RAG 请求按全文顺序生成

修改 `build-rag-request.py`（或新增 `build-rag-request-sequential.py`）：

每个 RAG 请求条目必须包含：

```json
{
  "order": 5,
  "paragraph_id": "p2",
  "sentence_id": "s3",
  "claim_id": "claim-005",
  "original_text": "平台规则的比例原则审查应当...",
  "claim_type": "policy_judgment",
  "citation_need": "critical",
  "citation_type": "secondary_source",
  "keywords": ["比例原则", "平台规则", "审查"],
  "chapter_context": "第二章 平台规则的合法性审查",
  "expected_evidence_type": "后设归纳",
  "search_dimensions": { ... }
}
```

#### 5.2.2 RAG 返回结果必须记录

每条 RAG 返回必须记录到 evidence-interpretations 中：

```json
{
  "claim_id": "claim-005",
  "order": 5,
  "paragraph_id": "p2",
  "sentence_id": "s3",
  "candidates": [
    {
      "candidate_id": "claim-005-cand-01",
      "retrieved_chunk_id": "chunk-abc-123",
      "chunk_text": "...",
      "source_ref_id": "ref-012",
      "item_key": "zotero-XYZ",
      "markdown_path": "/parsed/source.md",
      "page_map": {"page": 42},
      "support_strength": "strong_support",
      "confidence": 0.85,
      "risk_flags": [],
      "can_enter_note": true,
      "can_enter_background": true,
      "must_return_to_补库": false,
      "grounding_status": "full_markdown_grounding"
    }
  ],
  "best_strength": "strong_support",
  "risks": []
}
```

#### 5.2.3 逐段/逐句循环

RAG 反查不是一次性批量调用。设计为：

1. 按全文顺序遍历每个 paragraph。
2. 对每个 paragraph 中的每个 claim sentence，生成 RAG 请求。
3. 请求按 order 排列，确保全文顺序。
4. 返回结果按 order 写入 evidence-interpretations。
5. 如果某段/某句无 RAG 结果，标记为 `no_support_found`，进入 gap handoff。

#### 5.2.4 RAG 证据解释/裁判能力

当前 skill 不硬编码 RAG 解释模型。设计为可配置：

在 `config/rag-protocol.yaml` 中增加：

```yaml
rag_interpreter:
  model: "configurable"  # 不硬编码
  capabilities:
    - evidence_explanation
    - support_strength_judgment
    - risk_detection
    - claim_fit_assessment
  offline_mode: true  # 0.5.2-dev 不调用真实模型
  fixture_bypass: true
```

如果未来需要 gpt-5.5 或其他模型做证据解释，通过配置切换，不污染离线测试。

#### 5.2.5 RAG 反查边界

- RAG 反查不是写作，不生成正文。
- RAG 反查只做：证据定位、证据解释、风险判断。
- RAG 反查结果直接写入 evidence trace ledger。

## 六、Evidence Trace Ledger 设计（mimo-style）

### 6.1 定位

Evidence trace ledger 是 ReferenceFootnote 的独立能力，不依赖正文写作 skill 的 mimo。

它是：
- 按全文顺序排列的逐条证据链。
- 最终核查、风险清理、参考文献消费检查、后续反查的唯一顺序主线。
- 不能只输出分散的脚注表、参考文献表、风险表。

### 6.2 Ledger 条目结构

文件：`state/evidence-trace-ledger.json`

```json
{
  "version": "0.5.2-dev",
  "article_id": "article-001",
  "total_entries": 42,
  "entries": [
    {
      "order": 1,
      "paragraph_id": "p1",
      "sentence_id": "s1",
```

## 七、独立写作池能力设计

### 7.1 定位

ReferenceFootnote 独立复制"写作池式审查"能力，不依赖正文写作 skill。

写作池只用于：
1. 注释位置是否合适
2. 注释措辞是否必要
3. 正文 claim 与证据强度是否匹配
4. 是否应删除、移动、合并注释
5. 是否需要完整段落重写

### 7.2 限制

1. 不从零写正文。
2. 不替代正文写作。
3. 不使用推进池。
4. 不使用正文写作 skill 的任务目录。
5. 如需改正文，必须返回完整段落，Codex 不得抽句拼合。
6. 注释不是参考文献脚注，不能写无聊废话或不必要背景。

### 7.3 脚本设计

#### 7.3.1 `scripts/build-writing-pool-review-request.py`

输入：
- `state/insertion-plan.json`（或 `state/cleaned-insertion-plan.json`）
- `state/evidence-trace-ledger.json`
- `state/article-structure.json`

输出：`state/writing-pool-review-request.json`

逻辑：
1. 按全文顺序遍历 ledger 中所有有 note_id 的条目。
2. 对每个条目生成 review item，包含：
   - order
   - insertion_id
   - claim_id
   - paragraph_id
   - sentence_id
   - body_context（从 article-structure 中提取完整段落）
   - current_note_text
   - support_strength
   - grounding_status
   - known_risks
   - references_consumed
   - review_questions（根据 risk_flags 自动生成）

3. 输出 `execution_status: prepared_not_executed`。

#### 7.3.2 `scripts/apply-writing-pool-review-result.py`

输入：`state/writing-pool-review-result.json`（外部写作池返回）
输出：更新 `state/evidence-trace-ledger.json` 的 writing_pool_decision 字段

逻辑：
1. 验证 result schema。
2. 对每个 result item，更新 ledger 对应条目的 writing_pool_decision。
3. 如果 decision 是 `return_paragraph_for_rewrite`，标记为 blocking。
4. 如果 decision 是 `drop_note`，标记为需要在 S90 清理。

#### 7.3.3 `scripts/validate-writing-pool-review.py`

输入：`state/writing-pool-review-result.json` + `state/evidence-trace-ledger.json`
输出：`state/writing-pool-gate-result.json`

检查：
- 是否存在 `return_paragraph_for_rewrite` 未处理。
- 是否存在 `drop_note` 未进入风险清理。
- 是否存在 `move_note` 未回写插入计划。
- 是否存在 `analogy_only` 被写作池误标为 `keep` 且注释措辞暗示直接实证。

### 7.4 写作池审查规则

| 规则 | 说明 |
|------|------|
| analogy_only 不得输出暗示直接实证的注释 | 写作池必须检查注释措辞 |
| ownership_unverified 不得输出"已核实"口径 | 写作池必须检查注释是否暗示已核实 |
| 写作池不得从零写正文 | 只能审查已有注释和正文的契合性 |
| 如需改正文，只能返回完整段落 | Codex 不得抽句拼合 |
| 写作池不得升级证据强度 | 只能降级或保持 |

## 八、重跑与恢复设计

### 8.1 恢复场景

| 场景 | 恢复策略 |
|------|----------|
| RAG 结果更新 | 从 S50 重跑：更新 evidence-interpretations → S55 grounding → S60 rebuild ledger（保留已有 note/reference 决策作为参考） |
| 风险清理后删除注释 | S90 → S95 rebuild：同步删除 insertion-plan 中对应条目 + 检查参考文献消费 + 更新 ledger |
| 写作池要求整段重写 | S80 阻断 → 等待写作池返回完整段落 → 重跑 S80 → S85 → S90 |
| 补库完成 | 从 S45 重跑：apply-intake-completion → S50 逐段/逐句 RAG → S55 → S60 → ... |
| 用户要求重新裁剪参考文献 | 从 S70 重跑：保留 ledger 中已核证据链，只重新执行 footnote/reference planning |
| 文献无 PDF 或无可消费 Markdown | S85 risk inventory 标记为 `no_pdf_no_markdown` → S90 清理：移出最终参考文献或降级为 background |

### 8.2 恢复机制

1. 读取 `state/flow-status.json` 确定当前阶段。
2. 根据恢复场景确定回退目标阶段。
3. 从目标阶段开始重跑，保留之前阶段的产物。
4. Ledger 支持增量更新：只更新受影响的条目，不重建整个 ledger。
5. 每次重跑后更新 `flow-status.json`。

### 8.3 增量更新规则

| 操作 | Ledger 影响 |
|------|-------------|
| 删除注释 | 对应条目 note_id=null, note_text=null, final_decision="deleted_by_cleanup" |
| 降级注释 | 对应条目 support_strength 降级, note_text 更新 |
| 移动注释 | 对应条目 paragraph_id/sentence_id 更新 |
| 新增 RAG 结果 | 对应条目 rag_result_id/retrieved_chunk_id/support_strength 更新 |
| 删除参考文献 | 所有引用该 reference_id 的条目 reference_id=null |
| 补库完成 | 新增或更新 rag_result_id, support_strength, grounding_status |

## 附录 B：与正文写作 mimo 的区别

| 维度 | 正文写作 mimo | ReferenceFootnote evidence trace ledger |
|------|--------------|----------------------------------------|
| 目的 | 全程记录写作过程中的证据消费 | 全程记录参考文献补注过程中的证据定位和决策 |
| 触发 | 写作池推进时自动记录 | 每个阶段脚本显式更新 |
| 粒度 | 按段落/论点 | 按句子/claim |
| 依赖 | 正文写作 skill 内部 | ReferenceFootnote 独立，不依赖正文写作 |
| 输出 | 暗链，不直接交付 | 显式 JSON，作为交付包主文件之一 |
| 消费者 | 正文写作 guard | ReferenceFootnote final gate + 人工审查 |
