# ReferenceFootnote 复用检索入库 / 正文写作能力审查

结论：NEEDS_P1_FIX_BEFORE_GITHUB

## 总体判断

ReferenceFootnote 0.1.0-dev 的阶段机、边界声明和离线 fixture 框架已经成型，但对 `检索入库` 和 `正文写作` 两个既有 skill 的复用停留在"知道对方存在"的层面，缺少对真实交接产物矩阵、字段规范、证据类型体系、质量门禁脚本输出格式和第二大轮回流协议的结构化对齐。

当前 `references/search-intake-interface.md` 和 `references/writing-skill-interface.md` 各只有一句话，`docs/handoff-protocol.md` 只列出顶层字段名，`build-search-handoff.py` 的 request 结构缺少 `检索入库` 真实消费的关键字段，`build-delivery.py` 的 `handoff_to_writing.json` 缺少 writer 可直接消费的证据类型、来源角色、消费深度和格式校验兼容字段。

没有 P0 级阻断项（仓库结构完整、阶段机可跑、离线 gate 可过），但有多项 P1 级问题需要在 GitHub 第一版发布前修复，否则其他开发者或 Codex 无法仅凭 ReferenceFootnote 仓库理解如何与上下游 skill 正确对接。

---

## P0

无。当前仓库结构完整，阶段机可跑，离线 fixture 可过，不存在阻断发布的致命缺陷。

## P1

当前只有一句话。应补齐：
- `检索入库` 的正式职责边界（只做 1/2/2.5，到 `2.5 ready` 停）
- `step25 -> writer` 交接产物矩阵的完整字段列表
- `round2-step25 -> writer` 增量交接产物
- `A / B / C` 库语义摘要
- `kb-state.json`、`kb-routing-map.json`、`rag-import-summary.md`、`kb-usage-brief.md`、`hss-handoff-v2.md`、`zotero-reference-master.md`、`zotero-reference-citation-export.md`、`source-concentration-report.md`、`source-consumption-priority-map.md`、`kb-argument-function-brief.md` 的用途说明
- ReferenceFootnote 的 `search_intake_request` 如何映射到 `检索入库` 的 `round2-search-plan.md` 格式
- 第二大轮回流约束：回流是 gap 驱动补料，不是泛检
- `intake-completion` 返回时应携带的字段（`macro_round`、`gap_id`、`source_need`、`priority`、`kb_routing`、`pdf_status`、`import_status`）

### P1-2：`references/writing-skill-interface.md` 过于简化

当前只有一句话。应补齐：
- `正文写作` 的正式职责边界（只做第三步）
- writer 消费 `handoff_to_writing.json` 时期望的字段
- writer 的证据类型体系：师承证据、文本证据、后设归纳
- writer 的论断类型与可写强度矩阵
- writer 的来源角色体系：理论锚点、概念界定、学术史定位、关键争议、材料依据、方法参照、旁证补充、风险限定
- writer 的来源消费深度分级：深度消费 vs 浅要参考
- writer 的 `scan-citation-hygiene.py`、`scan-critical-claim-citations.py`、`validate-gbt7714-bibliography.py` 输出格式摘要
- writer 的 `gap-routing-table.md` 和 `round2-search-plan.md` 格式
- writer 的 `round2-source-readiness-report.md` 格式
- ReferenceFootnote 的 `handoff_to_writing.json` 应如何被 writer 的 R2-A1/A2 消费

### P1-3：`docs/handoff-protocol.md` 缺少结构化字段规范

当前只列出顶层字段名。应补齐：
- `search_intake_request.requests[]` 每条的完整字段定义（含 `macro_round`、`gap_id`、`source_need`、`priority`、`source_direction`、`minimum_requirement`、`ideal_requirement`、`search_strategy.databases`、`search_strategy.source_types`、`search_strategy.discipline`、`search_strategy.keywords_zh`、`search_strategy.keywords_en`、`search_strategy.author_hints`、`search_strategy.theory_hints`）
- `intake_completion.results[]` 每条的完整字段定义（含 `request_id`、`claim_id`、`status`、`sources_found`、`kb_routing`、`pdf_status`、`import_status`、`zotero_id`、`ref_metadata`）
- `handoff_to_writing.json` 的完整字段定义（含 `evidence_type`、`source_role`、`consumption_depth`、`claim_type`、`need_level`、`support_strength`、`risks`、`rewrite_suggestion`、`gbt7714_footnote`、`no_insert_reason`）

### P1-4：`build-search-handoff.py` 缺少关键字段

当前脚本生成的 request 缺少：
- `macro_round`（第几大轮）
- `gap_id`（对应 writer 的 gap 编号）
- `source_direction`（来源方向）
- `search_strategy.keywords_en`（英文关键词）
- `search_strategy.author_hints`
- `search_strategy.theory_hints`
- `search_strategy.constraints`

当前关键词提取逻辑是硬编码 if-in 匹配，应改为从 evidence-map 的 `search_dimensions` 字段透传。

### P1-5：`build-delivery.py` 的 `handoff_to_writing.json` 缺少 writer 可消费字段

当前只输出 `insertions`、`no_insert_zones`、`quality_status`。应补齐：
- `evidence_type`（师承证据 / 文本证据 / 后设归纳 / 经验材料 / 一手材料）
- `source_role`（理论锚点 / 概念界定 / 材料依据 / ...）
- `consumption_depth_suggestion`（深度消费 / 浅要参考）
- `claim_type` 和 `need_level` 透传
- `gbt7714_footnote`（已格式化的 GB/T 7714 脚注成品）
- `rewrite_suggestion` 结构化（当前只是字符串）
- `high_risk_unsupported`（从 evidence-map 透传）
- `unresolved_critical_claims`（从 evidence-map 透传）
- `existing_references_merge_status`（已有参考文献核验状态）

### P1-6：缺少 `intake-completion` 的结构化 fixture

当前 `templates/intake-completion.template.json` 只有顶层骨架，`results[]` 为空。应新增至少一个完整 fixture，包含：
- 多条 result，覆盖 `completed`、`partial`、`failed` 状态
- 每条 result 带 `kb_routing`、`pdf_status`、`import_status`、`zotero_id`、`ref_metadata`
- 对应的 `apply-intake-completion.py` 应验证这些字段

### P1-7：缺少证据类型与 writer 证据类型的映射说明

`docs/evidence-classification.md` 只定义了 support strength 和 risk flags，没有对接 writer 的三层证据类型（师承证据、文本证据、后设归纳）。应在该文件或新文件中补齐：
- ReferenceFootnote 的 evidence map 如何标注证据类型
- 证据类型如何影响 insertion plan 的 `requires_rewrite` 和 `rewrite_suggestion`
- 证据类型如何传递给 writer 的论断强度治理

## 遗漏能力与边界

### 应显式对接但当前缺失的

1. **GB/T 7714 格式校验**：`plan-footnotes.py` 的 `footnote_text()` 只有中文法学期刊格式，应同时支持 GB/T 7714 并与 writer 的 `validate-gbt7714-bibliography.py` 输出兼容
2. **已有参考文献导入与核验**：`insertion-plan.template.json` 有 `existing_references_verified` 字段但无实现逻辑
3. **来源集中度**：evidence map 可自然产出但未输出
4. **来源消费优先级**：evidence map 可自然产出但未输出
5. **argument function brief**：evidence map 可按论证功能分类但未输出
6. **no-insert zones 与 writer 改写建议的交接**：当前只输出 reason，不输出改写方向
7. **高风险无据句的处理路径**：当前只列出，不说明 writer 或人工如何消费
8. **RAG 平台输出格式**：当前 `rag-protocol.yaml` 只定义了 ReferenceFootnote 自己的协议，未说明与 hss-rag-scholarflow 或 hss-rag-platform 的格式映射

### 不需要对接的（确认边界正确）

1. ✅ 不需要直接运行 CNKI/WoS/Zotero/PDF/RAG
2. ✅ 不需要直接操作 `openclaw-cnki-takeover`
3. ✅ 不需要运行 writing-pool/advance-pool/mimo
4. ✅ 不需要探测 `localhost:22`
5. ✅ 不需要 staging/production 部署（0.1.0-dev）
